# Configuration: Workspace Config File

The optional configuration for a workspace is defined in a file such as <code>{createConfigLocationPath("tsFile", WORKSPACE_CONFIG_FILE_NAME, WORKSPACE_CONFIG_PACKAGE_JSON_KEY)}</code> in its directory or in its `package.json` file.

You can also use the root config to set workspace configs in bulk by using workspace patterns.
See the [**workspace pattern configs**](/config/workspace-pattern-configs) page for more information.

## Type Definition

<SyntaxHighlighter language="typescript">
  {WORKSPACE_CONFIG_TYPE}
</SyntaxHighlighter>

### Alias

**Aliases** are an alternate name or names for a workspace. See the [workspace aliases](/concepts/workspace-aliases) page for more information.

### Tags

**Tags** are simple strings that can be used to group workspaces together. They can be used
in [workspace patterns](/concepts/workspace-patterns).

### Default Inputs

The `defaultInputs` used to determine which files are considered inputs for the workspace, 
such as for [affected workspaces](/concepts/affected) resolution. These are applied to all 
scripts in the workspace that don't configure their own inputs (see below).

See the [workspace inputs](/config/workspace-inputs) page for more information.

### Scripts

A map of script names to config specific to any respective script.
Inline workspaces can use an [optional name](/concepts/inline-scripts#script-name) to be matched.

#### Order

The `scripts[name].order` option is used to set the [script execution order](/concepts/script-execution-order) for scripts.

#### Inputs

The `scripts[name].inputs` option is used to set the inputs for the script, such as for [affected workspaces](/concepts/affected) resolution.

See the [workspace inputs](/config/workspace-inputs) page for more information.

### Workspace Dependency Rules

Using the `rules.workspaceDependencies` field, you can define rules for which workspaces are allowed to be dependencies,
using `allowPatterns`, `denyPatterns`, or both.

[Workspace patterns](/concepts/workspace-patterns) are used to match workspaces.

`allowPatterns` defines the permitted subset of dependencies. `denyPatterns` forbids specific dependencies.
When both are present, `denyPatterns` further filters within the subset permitted by `allowPatterns`.
You can also use [negated](/concepts/workspace-patterns#negation) workspace patterns to exclude workspaces from the allowed or denied list.

For example, you may disallow dependencies with the [tag](/config/workspace#tags) "backend" for a frontend workspace with `denyPatterns: ["tag:backend"]`.

[Workspace dependencies](/concepts/workspace-dependencies) are explicitly defined in workspaces' `package.json` files.

You can use the `mergeWorkspaceConfig` function to merge multiple workspace configurations left to right, with each subsequent config taking precedence.
