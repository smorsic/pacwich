# Configuration: Overview

Configuration is set by JSON. Config files can be written in TypeScript, JavaScript, JSON, or JSONC.

See [Project Configuration](/config/project) and [Workspace Configuration](/config/workspace) for more information on the purpose of each field.

## Quick Examples

### Project Configuration

The [project configuration](/config/project) is for project-wide settings.

<SyntaxHighlighter language="typescript">
{`// path/to/your/project/${createConfigLocationPath("tsFile", PROJECT_CONFIG_FILE_NAME, PROJECT_CONFIG_PACKAGE_JSON_KEY)}

${createTsFileExample(exampleProjectConfigSimple1, "root")}`}
</SyntaxHighlighter>

### Workspace Configuration

[Workspace configurations](/config/workspace) are defined in the directory of a workspace.

<SyntaxHighlighter language="typescript">
  {`// path/to/your/project/workspace/${createConfigLocationPath("tsFile", WORKSPACE_CONFIG_FILE_NAME, WORKSPACE_CONFIG_PACKAGE_JSON_KEY)}

${createTsFileExample(exampleWorkspaceConfigSimple, "workspace")}`}
</SyntaxHighlighter>

## Config Locations / File Types

### Configuration locations

Configuration files can be read from the following location. The first to be resolved in this order will be used:
<ol className="config-location-list">
{CONFIG_LOCATION_TYPES.map(type => <li key={type}><b>{createConfigLocationDescription(type)}</b>: <code>{createConfigLocationPath(type, "pacwich.<name>", "<key>")}</code></li>)}
</ol>

### Value Resolution

Values related to configuration are resolved in the following order of precedence:
1. An explicit CLI or API argument/option passed to a method/function/command
2. A value provided in a configuration file
3. A value provided via an [environment variable](/config/env-vars), if applicable
4. The generic default value, if applicable

## Debugging Configuration

You can use the CLI command `pacwich config debug` to print all resolved JSON for configuration.

See the [CLI reference](/cli/commands#config-debug) for more information.

## Security and Performance of TypeScript/JavaScript configs

Keep in mind that TypeScript and JavaScript config files offer the most power and flexibility but both execute code and 
can affect `pacwich`'s performance if configs are slow to resolve.

### Performance

The average `pacwich` project that uses config straightforwardly should not experience performance issues due to TS/JS config,
unless very heavy processing or `async` logic is used within them. Keep in mind that slow config files will slow down all `pacwich` operations, including shell completions that are project-aware. 

If you happen to need dynamic configs that slow down `pacwich`, consider using a separate process to generate JSON/JSONC configs dynamically
to take the load off of `pacwich`'s resolution.

### Security

If you may be interacting with `pacwich` monorepos that are untrusted, you can set [environment variable](/config/env-vars) <code>{getUserEnvVarName("disableExecutableConfigsDefault")}</code>
to `true` to disable loading TS/JS config files altogether. There is also the CLI [global flag](/cli/global-options#disable-executable-configs) for this.
Of course, a malicious repository already has a large variety of attack vectors beyond just a `pacwich` config, but
this exists as an optional guard against arbitrary code execution.
