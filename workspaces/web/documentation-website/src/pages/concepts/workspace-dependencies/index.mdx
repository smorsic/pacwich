# Concept: Workspace Dependencies

## Overview

[Workspaces](/concepts/glossary#workspace) can depend on other workspaces and import/export code between them. This is a native capability of workspaces in Bun.

### Setting up Dependencies - Example

For example, this might be some workspace's `package.json` file, where `my-workspace-b` depends on `my-workspace-a`:

`my-workspace-a/package.json`:
<SyntaxHighlighter language="json">
  {JSON.stringify({
    "name": "my-workspace-a",
    "main": "index.ts"
  }, null, 2)}
</SyntaxHighlighter>

`my-workspace-b/package.json`:
<SyntaxHighlighter language="json">
    {JSON.stringify({
    "name": "my-workspace-b",
    "dependencies": {
      "my-workspace-a": "workspace:*"
    }
  }, null, 2)}
</SyntaxHighlighter>

Code in `my-workspace-a` can then import code from `my-workspace-b` like any other dependency.

`my-workspace-a/index.ts`:
<SyntaxHighlighter language="typescript">
  {`export const myFunction = () => {
  \ \ console.log("Hello from my-workspace-a");
}`}
</SyntaxHighlighter>

`my-workspace-b/index.ts`:
<SyntaxHighlighter language="typescript">
  {`import { myFunction } from "my-workspace-a";

myFunction();`}
</SyntaxHighlighter>

## Workspace Dependency Rules

You can set rules to allow or deny other workspaces as dependencies in a workspace's [configuration](/config/workspace),
set by [workspace patterns](/concepts/workspace-patterns).

Use `allowPatterns` to define which dependencies are permitted or `denyPatterns` to forbid specific ones. When both are present,
`denyPatterns` will further filter the subset already permitted by `allowPatterns`.

For example, you may disallow dependencies with the [tag](config/workspace#tags) "backend" for a frontend workspace with `denyPatterns: ["tag:backend"]`.

## Running Scripts in Dependency Order

When running scripts via the [CLI](/cli/commands#run-script) or [API](/api/reference#runscriptacrossworkspaces), workspaces can be run in dependency order.

This means that a workspace's script will wait until any workspaces it depends on have completed. 
This is similar to how Bun's [`--filter` flag](https://bun.com/docs/pm/filter#dependency-order) works, though for `bun-workspaces`,
this behavior is opt-in.

This can be useful in situations where one workspace depends on another workspace's output, such as a build script's output.

### CLI Example
<SyntaxHighlighter language="bash">
  {`bw run my-script --dep-order

# Run all scripts even if a dependency fails
bw run my-script --dep-order --ignore-dep-failure
  `}
</SyntaxHighlighter>

### API Example
<SyntaxHighlighter language="typescript">
  {`
import { createFileSystemProject } from "bun-workspaces";

const project = createFileSystemProject();

project.runScriptAcrossWorkspaces({
\ \ script: "my-script",
\ \ dependencyOrder: true,
\ \ ignoreDependencyFailure: true,
});
`}
</SyntaxHighlighter>
